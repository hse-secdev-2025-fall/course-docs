# Как устроен курс «Разработка безопасного ПО» - 1‑пейджер (2025)

## Модель работы
- **Один приватный репозиторий на студента на весь семестр** (P01…P14) через **GitHub Classroom** (assignment `course-project`).
- Каждая практика = **ветка** `pXX-*` → **PR «PXX - …»** → **ревью** → **зелёный CI** → **merge** → **тег `PXX`**.

## Что считаем «сдано»
- PR по шаблону, чек‑лист практики выполнен.
- Все статус‑чеки CI зелёные (ruff/black/isort/pytest/pre‑commit).
- Замечания **blocking** устранены; ревью запрошено у `@hse-secdev-2025-fall/instructors`.

## Оценка (коротко)
- Критерии по рубрикам (см. рубрику).
- Просрочка - по политике курса (см. `policies/40_grading-policy.md`).

## Коммуникации и артефакты
- Основной канал - **Issues/PR в вашем репозитории**.  
- Дополнительно: LMS/Telegram.  
- К финалу: теги `P01…P14`, итоговый отчёт и презентация (структура будет дана).

## Минимальные требования к проекту (ядро)
- Аутентификация и 1 CRUD‑ресурс.
- Тесты (позитив/негатив), линтеры/форматирование.
- Минимальный CI‑пайплайн.

## Почему так
- Сохраняем контекст угроз/контролей в **одном** месте.
- Не наращиваем «зоопарк» репозиториев.
- Учимся работать «как в проде»: ветки, PR, CI, теги, ревью.
